Disaster Recovery Runbook

This document outlines the procedures for recovering the X121 platform from various failure scenarios.

1. Full Database Restore

  1. Identify the most recent verified full backup from the admin dashboard.
  2. Stop all application services: systemctl stop x121-api x121-worker
  3. Restore the database from the backup file: 
    pg_restore -d x121_production -c -j 4 /path/to/backup.dump
  4. Apply any incremental WAL logs since the full backup.
  5. Run verification queries to confirm data integrity.
  6. Restart application services: systemctl start x121-api x121-worker

2. Point-in-Time Recovery (PITR)

  1. Identify the target recovery timestamp.
  2. Restore from the nearest full backup before the target time.
  3. Configure recovery_target_time in postgresql.conf.
  4. Replay WAL archives up to the target timestamp.
  5. Verify recovered data and promote the standby.

3. Configuration-Only Restore

  1. Download the latest config backup from the backup storage.
  2. Extract configuration files to the application config directory.
  3. Validate configuration: x121-api --validate-config
  4. Restart affected services.

4. Verification Procedure

  1. Check row counts for critical tables (users, projects, characters).
  2. Verify referential integrity across foreign keys.
  3. Run application health check endpoints.
  4. Confirm recent data (last 24h of activity logs) is present.
Warning: Always test restore procedures in a staging environment before applying to production.
Critical: Document the incident timeline and notify stakeholders within 15 minutes of initiating recovery.
